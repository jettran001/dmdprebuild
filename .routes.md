## 2. Ma trận Kết nối Module

| Module     | Common | Blockchain | Wallet | SnipeBot | AI   |
|------------|--------|------------|--------|----------|------|
| Common     | -      | Cao        | Cao    | Cao      | Trung bình |
| Blockchain | Cao    | -          | Cao    | Trung bình | Thấp  |

**Chú thích**:

- **Thấp**: Phụ thuộc tối thiểu, ít tương tác trực tiếp

## 3. Dependency Cycles và Giải pháp

**Chu trình phụ thuộc:** SnipeBot → Wallet → Blockchain → Common → SnipeBot

**Giải pháp**:
- Sử dụng dependency injection để giảm phụ thuộc trực tiếp
- Sử dụng event-driven architecture để giảm coupling
- Tách interfaces từ implementations (trait objects)
- Sử dụng Abstract Factory pattern cho việc tạo các service

```
┌─────────────┐      ┌─────────────┐
│             │──┬──►│             │
│  SnipeBot   │  │   │   Wallet    │
```

## 4. Use Case: Thực hiện giao dịch từ SnipeBot

```
┌───────────┐      ┌───────────┐      ┌───────────┐      ┌───────────┐      ┌───────────┐
│           │      │           │      │           │      │           │      │           │
│    API    │─────►│ SnipeBot  │─────►│  Wallet   │─────►│Blockchain │─────►│   Node    │
│           │      │           │      │           │      │           │      │           │
└───────────┘      └───────────┘      └───────────┘      └───────────┘      └───────────┘
      │                  ▲                  │                  │
      │                  │                  │                  │
      │                  │                  │
      │              ┌───┴───────┐          │                  │
      └──────────────►           │          │                  │
                     │    AI     │◄─────────┴──────────────────┘
                     │           │
                     └───────────┘

Quy trình:
1. API nhận yêu cầu thực hiện giao dịch
2. SnipeBot tạo tham số giao dịch
3. AI đề xuất timing và gas tối ưu
4. SnipeBot gửi yêu cầu ký giao dịch đến Wallet
5. Wallet ký và trả về giao dịch đã ký
6. SnipeBot gửi giao dịch đến Blockchain
7. Blockchain chuyển giao dịch đến Node
8. Blockchain theo dõi trạng thái giao dịch
9. AI cập nhật model dựa trên kết quả
10. SnipeBot trả về kết quả giao dịch cho API
```

### Luồng xử lý lỗi:

```
┌───────────┐      ┌───────────┐      ┌───────────┐      ┌───────────┐
│           │      │           │      │           │ 2.lỗi│           │
│    API    │◄─────┤ SnipeBot  │◄─────┤  Wallet   │◄─────┤Blockchain │
│           │5.thông│           │4.retry│           │3.retry│           │
└───────────┘  báo  └───────────┘      └───────────┘      └───────────┘

Quy trình xử lý lỗi:
1. Blockchain gặp lỗi kết nối với node
2. Blockchain áp dụng retry policy và trả về lỗi nếu quá số lần retry
3. Wallet nhận lỗi, áp dụng retry khác (ví dụ: đổi node)
4. SnipeBot nhận lỗi, áp dụng chiến lược phục hồi (điều chỉnh gas, timing)
5. API nhận thông báo lỗi và trạng thái phục hồi
```

### Feature Flags trong Common

Common module hỗ trợ các feature flags để bật/tắt các tính năng không cần thiết, được định nghĩa trong `Cargo.toml`:

```toml
[features]
default = ["subscription_disabled"]
subscription_disabled = []
```

#### Mô tả các feature flags:
- **subscription_disabled**: Vô hiệu hóa module subscription và các chức năng liên quan
- **default**: Danh sách các feature mặc định được bật khi không chỉ định rõ

#### Cách sử dụng feature flags:
1. Khi cần sử dụng module subscription, thêm vào `Cargo.toml` của dự án:
   ```toml
   [dependencies]
   common = { path = "../common", default-features = false }
   ```

2. Hoặc chỉ định rõ feature muốn sử dụng:
   ```toml
   [dependencies]
   common = { path = "../common", features = [] }
   ```

3. Trong code sử dụng conditional compilation để xử lý các tính năng tùy chọn:
   ```rust
   // Sử dụng conditional compilation để tùy chọn import module
   #[cfg(not(feature = "subscription_disabled"))]
   mod subscription;

   // Conditional re-export cho subscription module
   #[cfg(not(feature = "subscription_disabled"))]
   pub use subscription::{
       SubscriptionManager, 
       UserSubscription,
       // các struct và trait khác...
   };
   ```

### Server Router trong Common

#### Chi tiết Router trong `common/src/router.rs`

Module router quản lý việc định nghĩa, đăng ký và xây dựng API endpoints cho hệ thống. Các thành phần chính bao gồm:

```rust
/// Định nghĩa một route API
pub struct ApiEndpoint {
    /// Đường dẫn của endpoint
    pub path: String,
    /// Phương thức HTTP (GET, POST, PUT, DELETE)
    pub method: HttpMethod,
    /// Mô tả về endpoint
    pub description: String,
    /// Module sở hữu endpoint
    pub module: ModuleType,
    /// Handler cho endpoint
    pub handler: EndpointHandler,
    /// Có yêu cầu xác thực không
    pub require_auth: bool,
    /// Giới hạn rate limit (requests/phút)
    pub rate_limit: Option<u32>,
}

/// Module Type
pub enum ModuleType {
    Common,
    Blockchain,
    Wallet,
    SnipeBot,
    AI,
}
```

- Đăng ký endpoints mới
- Xây dựng router từ các endpoints đã đăng ký
- Lấy danh sách endpoints theo module
- Áp dụng middleware như authentication và rate limiting

#### Các patterns chính trong Server

1. **Server Trait Pattern**
```rust
/// Server trait định nghĩa các phương thức cần thiết cho một server
pub trait ServerTrait {
    fn new(addr: SocketAddr) -> Self;
    async fn start(&self) -> Result<()>;
    async fn stop(&self) -> Result<()>;
}
```

2. **State Management Pattern**
```rust
// Ví dụ về endpoint sử dụng state
async fn metrics(State(state): State<Arc<RwLock<ServerState>>>) -> String {
    let state = state.read().await;
    format!("connections {}\nrequests {}\n", state.connections, state.requests)
}
```

3. **Router Configuration Pattern**
```rust
// Tạo một Router không có state
let app_without_state = Router::new()
    .route("/health", get(|| async { "ok" }))
    .nest("/api", Router::new().route("/ping", get(|| async { "pong" })));

// Tạo router với state
let app_with_state = Router::new()
    .route("/health", get(health_check))
    .route("/metrics", get(metrics))
    .with_state(state.clone());
```

### Mô hình tích hợp Common Module

Common module được thiết kế để sử dụng trong các module khác thông qua các patterns sau:

1. **Dependency Injection Pattern**
```rust
// Inject RouteManager vào ServerGateway
let route_manager = Arc::new(router::RouteManager::new(route_config));
let server = server::GatewayServer::new(addr, route_manager);
```

2. **Middleware Integration Pattern**
```rust
// Router áp dụng middleware từ common
let router = Router::new()
    .route("/api/v1/users", get(user_handler))
    .layer(middleware::from_fn(logging_middleware))
    .layer(middleware::from_fn(auth_middleware));
```

3. **State Sharing Pattern**
```rust
// Chia sẻ state giữa nhiều modules
let app_state = Arc::new(RwLock::new(AppState::default()));
let blockchain_router = blockchain_routes(app_state.clone());
let wallet_router = wallet_routes(app_state.clone());
```

### API Export/Import Pattern

Cách exports API endpoints từ các module:

```rust
// Module: subscription.rs
pub fn subscription_routes() -> Router {
    Router::new()
        .route("/subscriptions", get(get_all_subscriptions))
        .route("/subscriptions/:id", get(get_subscription_by_id))
        .route("/subscriptions", post(create_subscription))
        .route("/subscriptions/:id", put(update_subscription))
        .route("/subscriptions/:id", delete(delete_subscription))
}

// Sử dụng trong module khác:
#[cfg(not(feature = "subscription_disabled"))]
use crate::subscription::subscription_routes;

async fn configure_routes(&self) -> Router {
    let base_routes = Router::new()
        .route("/health", get(health_check));

    #[cfg(not(feature = "subscription_disabled"))]
    let routes = base_routes.nest("/api", subscription_routes());
    
    #[cfg(feature = "subscription_disabled")]
    let routes = base_routes.nest("/api", Router::new());
    
    routes
}
```
